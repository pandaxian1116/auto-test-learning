## 一、软件测试基础

### 1. 软件测试 vs 程序调试

| 对比维度       | 软件测试           | 程序调试（Debug）    |
| -------------- | ------------------ | -------------------- |
| **目标** | **发现错误** | **改正错误**   |
| **阶段** | 测试阶段           | 测试后 / 开发中      |
| **动作** | 执行程序，观察结果 | 定位 → 修改 → 验证 |

#### 软件测试基本准则

- 所有测试追溯到需求
- 严格执行测试计划
- 注意群集现象（缺陷聚集）
- 程序员避免检查自己的程序
- 穷举测试不可能
- 妥善保存测试文档

#### 程序调试基本步骤

1. **错误定位**
2. **修改设计和代码**
3. **回归测试**（防止引入新错误）

---

### 2. 测试驱动开发（TDD）

**核心循环**：**红 → 绿 → 重构**（Red → Green → Refactor）

| 步骤                 | 说明                               |
| -------------------- | ---------------------------------- |
| 🔴**Red**      | 先编写一个**失败的测试用例** |
| 🟢**Green**    | 编写功能代码使其**通过测试** |
| 🔵**Refactor** | 重构代码以**优化设计**       |

> 💡 **记忆口诀**：先写测试再写码，红绿重构三步走。

---

### 3. 测试用例的关键属性

**定义**：测试用例是测试执行的最小单元，包含输入数据、执行步骤、预期结果等要素。好的测试用例应具备以下四个核心属性：

| 属性 | 英文 | 说明 | 重要性 |
|------|------|------|--------|
| **可复用性** | Reusability | 能够被重复使用，便于进行**回归测试**和**自动化测试** | 保证测试投入的性价比 |
| **可评估性** | Evaluability | 有明确的**预期结果和评估标准**，能够客观判断测试是否通过 | 确保测试结果的准确性 |
| **有效性** | Effectiveness | 能够**有效地发现缺陷**，覆盖关键功能点和边界条件 | 体现测试的价值 |
| **易组织性** | Organizability | 便于**管理和维护**，包括合理的分类组织、优先级划分、版本控制 | 提升测试的可维护性 |

#### 详细说明

**A. 可复用性（Reusability）**
- 测试用例应避免过于特殊化和一次性使用
- 设计时要考虑到复用的可能性
- 好的可复用性可大幅降低回归测试成本

**B. 可评估性（Evaluability）**
- 必须包含：输入数据、执行步骤、预期输出
- 评估标准要清晰明确，无歧义
- 确保不同测试人员执行结果一致

**C. 有效性（Effectiveness）**
- 能够有效发现软件中存在的缺陷
- 覆盖关键功能点和边界条件
- 避免无效或重复的测试用例浪费资源

**D. 易组织性（Organizability）**
- 合理的分类组织（按模块、功能、优先级等）
- 优先级划分（P0 ~ P3）
- 版本控制和维护更新

> 💡 **记忆口诀**：测试用例四属性 = **可复用 + 可评估 + 有效 + 易组织**，缺一不可

---

### 4. 测试优先级策略

在软件测试过程中，**选择风险最大的场景进行测试覆盖是最有效的方法**，原因：

1. 优先测试风险较大的场景可以尽早发现系统中最关键的问题
2. 风险场景往往涉及核心功能或重要业务流程，测试投入产出比最高
3. 符合软件测试中的**二八原则** — 80% 的缺陷出现在 20% 的代码中
4. 有利于合理分配有限的测试资源，将重点放在最需要关注的地方

## 二、测试方法（黑盒/白盒/灰盒）

### 1. 白盒测试（White-box Testing）

**定义**：测试人员**完全了解**程序内部结构和代码逻辑，基于代码实现设计测试用例。

#### 核心关注点

- **代码覆盖率**：语句覆盖、分支覆盖、条件覆盖、路径覆盖、MC/DC 覆盖
- **控制流分析**：循环复杂度、嵌套深度
- **数据流分析**：变量定义-使用链（def-use chain）

#### 常用技术

- 单元测试（JUnit、pytest 等）
- 静态代码分析（SonarQube、ESLint）
- 路径测试、边界值分析（基于代码内部边界）

#### 优势

- 能发现隐藏的逻辑错误和死代码
- 精确评估测试充分性（覆盖率指标）
- 便于定位缺陷根因

#### 局限性

- 无法验证需求正确性（只能验证实现正确性）
- 成本高，需要测试人员具备开发能力
- 无法覆盖所有路径（路径爆炸问题）

---

### 2. 灰盒测试（Gray-box Testing）

**定义**：介于黑盒测试和白盒测试之间，测试人员**部分了解**系统内部结构（如数据库结构、API 接口定义、架构文档），但不需要像白盒测试那样深入代码实现细节。

#### 核心特点

- 基于对系统内部有限认知设计测试用例
- 结合输入/输出验证与内部状态检查
- 关注数据流、控制流和异常处理路径

#### 常用技术

- 基于 API 文档的接口测试
- 基于数据库结构的 SQL 注入测试
- 基于日志分析的错误定位
- 基于配置文件的状态转换测试

#### 优势

- 比黑盒测试更有针对性，能发现更深层次的缺陷
- 比白盒测试门槛更低，不需要精通代码
- 适合微服务架构下的集成测试

#### 典型应用场景

- 微服务间的接口联调测试
- 基于 Swagger 文档的 API 自动化测试
- 性能瓶颈定位（结合监控指标）

---

### 3. 三种测试方法对比总结

| 维度                       | 黑盒测试             | 灰盒测试                  | 白盒测试           |
| -------------------------- | -------------------- | ------------------------- | ------------------ |
| **对内部结构的了解** | 无                   | 部分（接口/架构/配置）    | 完全（代码级）     |
| **测试依据**         | 需求规格说明         | 设计文档 + 部分代码       | 源代码             |
| **典型人员**         | 功能测试工程师       | 测试开发 / 接口测试工程师 | 开发工程师         |
| **主要目标**         | 验证功能是否符合需求 | 验证模块交互和数据一致性  | 验证代码逻辑正确性 |
| **常用工具**         | Postman、Selenium    | Charles、Swagger、JMeter  | JUnit、JaCoCo、GDB |

### 4. 测试方法适用性（黑白盒对照表）

| 方法               | 黑盒 | 白盒 | 说明                                                |
| ------------------ | :--: | :--: | --------------------------------------------------- |
| **边界值法** |  ✅  |  ✅  | **唯一黑白盒通用**，输入域/内部逻辑边界均适用 |
| 逻辑覆盖法         |  ❌  |  ✅  | 语句覆盖、分支覆盖等代码逻辑测试                    |
| 基本路径法         |  ❌  |  ✅  | 程序控制流程测试                                    |
| 正交实验设计法     |  ✅  |  ❌  | 多参数组合优化，大规模输入场景                      |

> 💡 **记忆要点**：
>
> - **唯一黑白盒通用**：边界值法
> - **纯白盒**：逻辑覆盖、基本路径
> - **纯黑盒**：正交实验设计

### 5. 白盒测试覆盖率详解

#### 覆盖率强度排序

```
```

```
语句覆盖 < 判定覆盖 < 条件覆盖 < 判定/条件覆盖 < 条件组合覆盖 < 路径覆盖
```

| 覆盖标准                       | 说明                           |     强度     |
| ------------------------------ | ------------------------------ | :----------: |
| **语句覆盖**             | 每个可执行语句至少执行一次     |      ⭐      |
| **判定覆盖（分支覆盖）** | 每个判定的真假分支至少执行一次 |     ⭐⭐     |
| **条件覆盖**             | 每个条件的真假取值至少出现一次 |    ⭐⭐⭐    |
| **判定/条件覆盖**        | 同时满足判定覆盖和条件覆盖     |   ⭐⭐⭐⭐   |
| **条件组合覆盖**         | 所有条件取值的组合至少出现一次 |  ⭐⭐⭐⭐⭐  |
| **路径覆盖**             | 程序所有可能的路径至少执行一次 | ⭐⭐⭐⭐⭐⭐ |

> **注意**：条件覆盖与判定覆盖**无严格包含关系**。

#### MC/DC 覆盖（最严格的工程标准）

**定义**：要求每个条件**独立地影响**判定结果，即对于每个条件，存在一对测试用例，只有该条件不同，且判定结果也不同。

**特点**：

- 同时包含条件覆盖和判定覆盖
- 被航空航天标准 **DO-178C** 要求在 **A 级（最高安全等级）** 软件中使用
- 是实际工程中**要求最严格**的覆盖率标准之一

## 三、测试类型详解

### 1. 测试类型层级

| 测试类型           | 关注重点                     | 执行者                |
| ------------------ | ---------------------------- | --------------------- |
| **单元测试** | 代码单元逻辑                 | 开发                  |
| **集成测试** | 模块间接口                   | 开发 / 测试           |
| **系统测试** | 整体系统功能                 | 测试团队              |
| **验收测试** | **业务需求、用户期望** | **用户 / 客户** |

---

### 2. 功能测试（Functional Testing）

**定义**：系统测试的主要内容之一，主要目的是验证软件的**各个功能是否按照需求规格说明**正常工作。

**与验收测试的区别**：

| 对比维度 | 功能测试 | 验收测试 |
|---------|---------|---------|
| **执行阶段** | 系统测试阶段 | 系统测试之后 |
| **执行者** | 测试团队 | 用户 / 客户 |
| **依据** | 需求规格说明书 | 业务需求、用户期望 |
| **目标** | 验证功能是否正确实现 | 验证是否满足业务需求 |

---

### 3. 验收测试（Acceptance Testing）

**定义**：验证软件系统**是否符合业务需求和用户期望**，通常由**用户或客户**执行。

---

### 5. 集成测试方法

#### ① 非渐增组装测试（非增式集成测试）

- 将单元测试后的模块**一次性**按照总体结构图集成起来
- 一般用**黑盒法**编写测试集
- **缺点**：程序错误易出现，不容易定位问题
- **适用**：规模小的开发系统

#### ② 渐增组装测试（增式集成测试）

- 在单元测试的基础上，**逐层安装测试**（自顶向下 / 自底向上）
- 可将错误分解，**容易定位**问题
- **适用**：大规模的开发系统

---

### 6. 测试类型选择（支付回调场景）

**场景**：验证服务**解析支付成功回调并更新数据库**。

| 维度                   | A 端到端           | B 单元测试                    | **C 集成测试** ✅                | D 契约测试 |
| ---------------------- | ------------------ | ----------------------------- | -------------------------------------- | ---------- |
| **效率**         | 低（完整流程）     | 高                            | **较高**                         | 中         |
| **可靠性**       | 不稳定（依赖外部） | 不足（全 mock，遗漏集成问题） | **好（真实数据库）**             | 一般       |
| **隔离性**       | 差                 | 好                            | **强（测试容器 + mock server）** | 强         |
| **核心逻辑覆盖** | 间接               | 不完整                        | **直接验证解析 + 数据库更新**    | 不直接     |

**结论**：**C（集成测试）** 综合最佳：隔离的真实数据库 + mock 支付网关，兼顾效率、可靠性和隔离性。

---

### 7. 决策表法（Decision Table Testing）

**定义**：处理**多个条件组合规则**的最有效测试设计方法，通过表格形式列出所有条件及其对应的动作。

**适用场景**：
- 多个输入条件之间存在逻辑组合关系
- 不同条件组合导致不同的处理结果
- 业务规则复杂，需要系统化覆盖

**与其他黑盒方法对比**：

| 方法 | 适用场景 | 特点 |
|------|---------|------|
| **决策表法** ✅ | **多个条件组合规则** | 最有效处理条件组合 |
| 等价类划分 | 输入数据分类 | 减少测试用例数量 |
| 边界值分析 | 边界条件验证 | 发现边界缺陷 |
| 正交实验法 | 大规模参数组合 | 优化组合数量 |

> 💡 **记忆要点**：看到**多个条件组合规则** → 选**决策表法**

---

### 8. 压力测试（Stress Testing）

**定义**：专门用来评估系统在**资源耗尽或极限负载情况**下的表现的测试方法。

**特点**：通过模拟极端条件（如内存不足、磁盘空间耗尽、网络带宽受限等）来测试系统的稳定性和容错能力。

---

### 9. 突变测试（Mutation Testing）

**定义**：通过向源代码中**引入小的变化（突变）**，检查现有测试用例能否检测出这些变化。

#### 关键指标：突变分数

- **85% 突变分数** = 现有测试用例能检测并"杀死"85% 的代码变异
- 剩余 **15% 存活突变体** = 测试覆盖不足，**不代表源代码有缺陷**

#### 常见误区

| 错误理解              | 纠正                                          |
| --------------------- | --------------------------------------------- |
| 预测未来缺陷          | ❌ 只评估当前测试质量                         |
| 存活突变体 = 代码缺陷 | ❌ 突变体是人为引入，非原始代码问题           |
| 等同于代码覆盖率      | ❌ 突变分数衡量测试有效性，覆盖率衡量测试范围 |

---

### 10. 测试替身（Test Double）

**定义**：在单元测试中，用**模拟对象**替代真实依赖（如数据库、外部 API），以便隔离被测代码进行独立验证。

#### 常见测试替身类型

| 类型 | 英文 | 说明 | 适用场景 |
|------|------|------|---------|
| **桩（Stub）** | Stub | 提供预设的返回值，不验证调用行为 | 需要依赖返回特定数据 |
| **模拟（Mock）** | Mock | **记录方法调用次数、参数**，通过断言进行行为验证 | 验证交互行为是否正确 |
| **伪对象（Fake）** | Fake | 轻量级的真实实现替代（如内存数据库） | 需要真实逻辑但不想用真实服务 |
| **哑元（Dummy）** | Dummy | 仅作为参数填充，不被实际使用 | 方法签名需要但未使用的参数 |
| **间谍（Spy）** | Spy | 对真实对象的包装，记录调用信息 | 在真实对象基础上增加监控 |

#### 面试高频场景：Mock 的选择

**场景描述**：在单元测试中需要验证——当输入有效时，Service 应调用 `Repository.save` **且仅调用一次**，并校验传入的参数。

**答案**：应优先选用 **Mock（模拟）**

**解析**：
- ✅ Mock 可以**记录方法调用次数**（验证 `save` 是否被调用了 1 次）
- ✅ Mock 可以**捕获传入的参数**（验证传入的参数是否正确）
- ✅ Mock 支持在测试中通过**断言进行行为验证**（`verify` 方法）
- ❌ Stub 只能返回预设值，无法验证调用行为
- ❌ Fake 虽然可用，但无法精确验证调用次数和参数

**代码示例（伪代码）**：
```java
// 创建 Mock
Repository mockRepo = mock(Repository.class);
Service service = new Service(mockRepo);

// 执行测试
service.process(inputData);

// 验证行为：save 被调用且仅调用一次，参数为 inputData
verify(mockRepo, times(1)).save(inputData);
```

> 💡 **记忆要点**：需要验证**行为（调用次数、参数）** → 选 **Mock**；只需要**返回值** → 选 **Stub**

## 四、性能测试

### 1. 性能测试范围

#### ✅ 属于性能测试

- 响应时间
- 资源利用率（CPU / 内存 / 磁盘 / 网络）
- 吞吐量（TPS / QPS）

#### ❌ 不属于性能测试

- **数值检测** — 属于功能测试 / 数据校验范畴

---

### 2. 性能测试类型对比

| 测试类型                   | 负载水平         | 持续时间                      | 核心目标                   |
| -------------------------- | ---------------- | ----------------------------- | -------------------------- |
| **浸泡测试（Soak）** | 正常负载         | **长时间（小时 ~ 天）** | **稳定性、资源泄漏** |
| 压力测试（Stress）         | **超负载** | 短期                          | 崩溃点、容错行为           |
| 容量测试（Capacity）       | 递增至极限       | 短期 ~ 中期                   | 最大处理能力               |
| 负载测试（Load）           | 正常负载         | **短期**                | 响应时间、吞吐量           |

> 💡 **关键记忆点**：
>
> - **长时间 + 正常负载 + 资源泄漏** = 浸泡测试
> - 看到 **"72小时"、"内存泄漏"、"持续运行"** 选 **Soak / Endurance Testing**

---

### 3. 疲劳强度测试（Soak Testing）

#### 最先发现的问题

1. **内存泄露**
2. **内存不足**

> 长时间运行下，资源泄漏问题最先暴露。

---

### 4. JMeter 阶梯线程组（Stepping Thread Group）

**定义**：JMeter 插件，可以**精确控制用户数的阶梯式增长**，例如每 30 秒增加 100 个用户，适合逐步加压场景。

| 组件                               | 功能               | 特点               |
| ---------------------------------- | ------------------ | ------------------ |
| **Stepping Thread Group** ✅ | 阶梯式增加并发用户 | 控制粒度精细       |
| 普通 Thread Group                  | 线性增长           | 控制粒度粗         |
| Constant Throughput Timer          | 控制吞吐量         | 非并发数控制       |
| Synchronizing Timer                | 集合点             | 让线程同时触发请求 |

## 五、部署策略

### 1. 蓝绿部署（Blue-Green Deployment）

**核心原理**：同时维护**两套完全相同的生产环境**，通过**流量切换**完成版本更替。

```
┌─────────────┐     ┌─────────────┐
│  蓝色环境    │     │  绿色环境    │
│ （当前版本）  │     │  （新版本）   │
│  ← 100%流量  │     │              │
└─────────────┘     └─────────────┘
         ↓ 切换流量
┌─────────────┐     ┌─────────────┐
│  蓝色环境    │     │  绿色环境    │
│ （回滚备份）  │     │  （新版本）   │
│              │     │  ← 100%流量  │
└─────────────┘     └─────────────┘
```

**部署流程**：

1. 蓝色环境运行当前版本（100% 流量）
2. 在绿色环境部署新版本，进行验证
3. 将流量从蓝色切换到绿色（**瞬间完成**）
4. 保留蓝色环境作为**回滚备份**
5. 观察稳定后，释放蓝色环境资源

**关键特征**：

- ✅ **零停机时间**：流量切换是原子操作
- ✅ **即时回滚**：发现问题可立即切回蓝色环境
- ❌ **资源成本高**：需要双倍基础设施资源

**适用场景**：

- 金融交易系统（不能容忍任何停机）
- 需要严格合规审计的环境（保留完整旧版本）
- 资源充足且对稳定性要求极高的核心业务

**注意事项**：

- 数据库 schema 变更需要兼容（或采用扩展-收缩模式）
- 会话保持问题（切换时用户会话需同步或支持无状态）
- 成本敏感场景需谨慎使用

---

### 2. 金丝雀部署（Canary Deployment）

**核心原理**：将**少量用户流量**先路由到新版本（金丝雀组），观察监控指标，逐步扩大流量比例，最终全量发布。

> 🐦 **命名来源**：矿工用金丝雀检测矿井瓦斯，先放少量金丝雀，安全后再让人进入。

**部署流程**：

```
新版本部署到少量节点（5% 流量）
        ↓ 监控指标正常
扩大流量比例（10% → 25% → 50% → 100%）
        ↓ 指标异常
自动回滚，仅影响少量用户
```

**关键特征**：

- **渐进式发布**：风险可控，影响范围小
- **真实环境验证**：在生产环境用小流量验证
- **A/B 测试基础**：可基于用户特征路由（地域、设备、用户 ID）

**适用场景**：

- 新功能效果验证（需要真实用户反馈）
- 大规模用户产品的谨慎发布
- 与 A/B 测试结合进行灰度放量

**监控重点**：

- **黄金指标**：延迟（Latency）、流量（Traffic）、错误（Errors）、饱和度（Saturation）
- **业务指标**：转化率、点击率、留存率

---

### 3. 滚动更新（Rolling Update）

**核心原理**：**逐个（或逐批）替换**旧版本实例为新版本实例，始终保持部分实例可用。

**部署流程**（假设 10 个实例）：

1. 拉出 2 个旧实例，停止流量
2. 在这 2 个节点部署新版本，健康检查后接入流量
3. 重复步骤 1-2，直到所有实例更新完成
4. 过程中始终有 **8 个实例提供服务**

**关键参数**：

- `maxSurge`：更新过程中可超出的实例数（如 +1 表示可多启动 1 个新实例）
- `maxUnavailable`：更新过程中允许不可用的最大实例数（如 1 表示最多 1 个不可用）

**关键特征**：

- ✅ **资源利用率高**：不需要额外全套环境
- ⚠️ **渐进式但非隔离**：新旧版本同时运行，存在兼容性问题
- ❌ **回滚复杂**：需要重新逐个替换回去

**适用场景**：

- 资源受限，无法承担蓝绿部署的双倍成本
- 无状态服务（微服务、Web 应用）
- 对停机时间有一定容忍度（秒级）

**注意事项**：

- **兼容性要求**：新旧版本必须能共存（数据库兼容、API 兼容）
- **长连接处理**：WebSocket、TCP 长连接可能被中断
- **状态服务慎用**：有状态服务（如缓存、数据库主节点）不适用

---

### 4. 功能开关（Feature Toggle / Feature Flag）

**核心原理**：将**代码部署**与**功能发布**解耦。新功能代码随版本一起部署到生产环境，但通过配置开关控制功能是否对用户可见。

**架构模式**：

```java
if (featureFlags.isEnabled("new-payment-flow", userContext)) {
    // 新功能逻辑
} else {
    // 旧功能逻辑
}
```

**开关类型**：

| 类型               | 说明                           | 示例                    |
| ------------------ | ------------------------------ | ----------------------- |
| **发布开关** | 控制功能是否上线               | 新支付渠道是否开放      |
| **实验开关** | A/B 测试，不同用户看到不同版本 | 新 UI 界面 50% 用户可见 |
| **运维开关** | 应对系统负载的熔断 / 降级      | 大促时关闭非核心功能    |
| **权限开关** | 基于用户角色 / 付费等级        | VIP 用户提前体验新功能  |

**部署流程**：

1. 开发完成新功能，代码中嵌入开关判断
2. 部署到生产环境（**默认关闭**，用户无感知）
3. 内部测试 / 小范围验证后，打开开关
4. 全量发布或逐步放量
5. 功能稳定后，清理旧代码和开关（技术债管理）

**关键特征**：

- **部署 ≠ 发布**：代码上线但功能不上线
- **快速回滚**：发现问题只需改配置，无需重新部署
- **主干开发（Trunk-based Development）基础**：支持频繁合并到主干

**适用场景**：

- 持续交付 / 持续部署（CI/CD）流水线
- 需要频繁发布但功能未完全就绪
- 大功能拆解为小迭代交付
- 紧急情况下快速关闭问题功能（熔断）

**注意事项**：

- **技术债风险**：开关过多会导致代码复杂度上升，需定期清理
- **测试复杂度**：需要测试开关开启和关闭两种状态
- **数据兼容性**：开关切换时涉及的数据格式需兼容

---

### 5. 四种部署策略对比

| 维度                   | 蓝绿部署                 | 金丝雀部署                    | 滚动更新                  | 功能开关                  |
| ---------------------- | ------------------------ | ----------------------------- | ------------------------- | ------------------------- |
| **资源成本**     | 🟥 高（双倍资源）        | 🟨 中（少量额外资源）         | 🟩 低（几乎无额外资源）   | 🟩 低（代码已部署）       |
| **回滚速度**     | 🟩 极快（秒级切换）      | 🟩 快（停止放量即可）         | 🟥 慢（需重新替换实例）   | 🟩 极快（改配置即可）     |
| **风险范围**     | 全量切换（但可快速回退） | 🟩 最小（仅影响少量用户）     | 🟨 中（新旧共存期间）     | 🟩 可控（精确到功能级别） |
| **停机时间**     | 🟩 零                    | 🟩 接近零                     | 🟨 接近零（但可能有瞬断） | 🟩 零                     |
| **适用场景**     | 核心系统、合规要求       | 新功能验证、A/B 测试          | 资源受限、无状态服务      | 持续交付、功能灰度        |
| **数据库兼容性** | 需兼容或扩展-收缩        | 需兼容                        | 必须兼容                  | 必须兼容                  |
| **复杂度**       | 🟨 中（需维护双环境）    | 🟥 高（需精细流量控制和监控） | 🟨 中（需处理共存兼容）   | 🟥 高（需开关管理系统）   |

## 六、面试高频题与代码分析

### 1. 异常捕获输出题

**题目**：`AccessViolationException` 异常触发后，下列程序的输出结果为？

```csharp
static void Main(string[] args)
{
    try
    {
        throw new AccessViolationException();
        Console.WriteLine("error1");
    }
    catch (Exception e)
    {
        Console.WriteLine("error2");
    }
    Console.WriteLine("error3");
}
```

**答案**：

```
error2
error3
```

**解析**：

- `throw` 后代码不执行，所以 `error1` 不会输出
- 异常被 `catch` 捕获，输出 `error2`
- `catch` 块后继续执行，输出 `error3`

### 2. MC/DC 覆盖测试用例设计

**题目**：用白盒测试方法测试以下代码片段，最多需要设计多少个测试用例？

```c
int func(Boolean A, Boolean B, Boolean C)
{
    if (A && (B || C))
        return 1;
    return 0;
}
```

**答案**：**4 个测试用例**

| 用例 | A | B | C | 结果 | 说明                               |
| :--: | :-: | :-: | :-: | :--: | ---------------------------------- |
|  1  | T | F | T |  1  | 条件 A 的 T 用例                   |
|  2  | F | F | T |  0  | 条件 A 的 F 用例（A 独立影响结果） |
|  3  | T | T | F |  1  | 条件 B 的 T 用例（B 独立影响结果） |
|  4  | T | F | F |  0  | 条件 C 的 F 用例（C 独立影响结果） |

**解析**：

- 对于条件 **A**：用例 1 和 2，A 取值相反，B 和 C 相同，判定结果分别为 1 和 0
- 对于条件 **B**：用例 1 和 3，B 取值相反，A 和 C 相同，判定结果分别为 1 和 0
- 对于条件 **C**：用例 3 和 4，C 取值相反，A 和 B 相同，判定结果分别为 0 和 1

### 3. 语句覆盖测试用例设计

**题目**：使用语句覆盖对下列代码进行测试用例设计时，需要设计多少个测试用例？

```c
public static void Test(int a, int b, int c)
{
    if ((a > 1) && (c == 0))
        b = b / a;
    if ((a == 1) && (b > 1))
        b = b + 1;
}
```

**答案**：**2 个测试用例**

| 用例 | a | b | c | 覆盖路径                       |
| :--: | :-: | :-: | :-: | ------------------------------ |
|  1  | 2 | 4 | 0 | 进入第一个 if，不进入第二个 if |
|  2  | 1 | 2 | 1 | 不进入第一个 if，进入第二个 if |

**解析**：语句覆盖只需要每个可执行语句至少执行一次。两个 `if` 分支中的赋值语句各需执行一次，因此最少需要 2 个用例。

### 4. 白盒测试概念题

**题目**：在白盒测试中，分支覆盖（也称判定覆盖）要求什么？

**答案**：要求程序中的**每个判定的真假分支**（即每个决策点的 true 和 false 结果）都**至少执行一次**。

### 5. 测试优先级策略题

**题目**：在软件测试过程中，为了提高测试效率和缩短测试周期，选择什么场景进行测试覆盖是最有效的方法？

**答案**：选择**风险最大的场景**进行测试覆盖是最有效的方法。

**原因**：

1. 优先测试风险较大的场景可以尽早发现系统中最关键的问题
2. 风险场景往往涉及核心功能或重要业务流程，测试投入产出比最高
3. 符合软件测试中的二八原则 — 80% 的缺陷出现在 20% 的代码中
4. 有利于合理分配有限的测试资源，将重点放在最需要关注的地方

### 6. Postman 接口测试题

**题目**：使用 Postman 去测试接口，以下说法正确的是？

**答案**：以下说法均正确：

| 操作 | Postman 实现方式 | 说明 |
|-----|----------------|------|
| **测试步骤** | 设置请求方式 → 输入请求路径 → 设置请求数据 → 点击发送请求 → 查看响应数据 | 标准测试流程 |
| **断言状态码** | `Status code: Code is 200` | 断言响应状态码是否为 200 |
| **获取 JSON 响应** | `var jsonData = pm.response.json()` | 获取 JSON 格式的响应数据 |
| **设置环境变量** | `pm.environment.set("key", "value")` | 设置环境变量 |
| **设置全局变量** | `pm.globals.set("key", "value")` | 设置全局变量 |

**解析**：

- ✅ **测试步骤**：设置请求方式（GET/POST 等）→ 输入 URL → 设置 Body/Headers → 点击 Send → 查看响应
- ✅ **状态码断言**：使用 `Status code: Code is 200` 可断言响应状态码是否为 200
- ✅ **获取 JSON 数据**：`pm.response.json()` 可获取并解析 JSON 格式的响应数据
- ✅ **环境变量**：`pm.environment.set()` 用于设置环境级别变量
- ✅ **全局变量**：`pm.globals.set()` 用于设置全局级别变量

> 💡 **记忆要点**：Postman 测试流程 = **请求设置 → 发送 → 断言验证**；变量作用域：**环境变量 < 全局变量**
