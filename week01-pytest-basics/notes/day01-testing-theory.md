## 1. 软件测试定义

> 测试是为了发现错误而执行程序的过程。——Glenford Myers（1979）

**核心观点**：
- 测试只能证明缺陷**存在**，不能证明缺陷**不存在**（Dijkstra定律）
- 好的测试用例：能发现至今未发现的错误
- 测试的目的：**质量保障**，而非"证明软件好用"

**软件测试的七大原则**（ISTQB）：

| # | 原则 | 说明 |
|---|------|------|
| 1 | 测试显示缺陷存在 | 测试可以证明缺陷存在，但不能证明无缺陷 |
| 2 | 穷尽测试是不可能的 | 除非软件极简，否则不可能测试所有路径 |
| 3 | 测试应尽早介入 | 需求阶段就应参与，越早发现缺陷成本越低 |
| 4 | 缺陷集群性 | 80%的缺陷集中在20%的模块（帕累托原则） |
| 5 | 杀虫剂悖论 | 反复执行相同测试，发现缺陷的能力下降 |
| 6 | 测试依赖上下文 | 安全关键系统与电商系统的测试策略不同 |
| 7 | 无错谬误 | 修复了缺陷但软件仍不满足需求，质量仍有问题 |

## 2. 软件测试生命周期（STLC）

软件测试生命周期（Software Testing Life Cycle）与软件开发生命周期并行推进：

```
需求分析 → 测试计划 → 测试设计 → 测试环境搭建 → 测试执行 → 测试关闭
```

### 各阶段详解

| 阶段 | 主要活动 | 进入条件 | 退出条件 | 产出物 |
|------|----------|----------|----------|--------|
| **需求分析** | 理解需求、识别测试范围、可行性评估 | 需求文档可用 | 需求理解完成 | 需求追踪矩阵 (RTM) |
| **测试计划** | 制定测试策略、资源评估、风险分析 | 需求分析完成 | 测试计划审批通过 | 测试计划文档 |
| **测试用例设计** | 编写测试用例、测试脚本、测试数据 | 测试计划审批 | 用例评审通过 | 测试用例、测试脚本 |
| **测试环境搭建** | 部署测试环境、准备测试数据 | 测试用例就绪 | 环境 Smoke Test 通过 | 测试环境、测试数据 |
| **测试执行** | 执行用例、记录缺陷、回归测试 | 环境就绪 | 退出标准满足 | 缺陷报告、测试报告 |
| **测试关闭** | 分析测试结果、总结经验、归档文档 | 测试执行完成 | 利益相关方确认 | 测试总结报告 |

## 3. V 模型（V-Model）

V模型是最经典的测试模型，左侧为开发阶段，右侧为测试阶段，每个开发阶段都对应一个测试阶段：

```
需求分析    ←——————————→   验收测试（UAT）
    ↓                          ↑
  系统设计  ←——————————→   系统测试
      ↓                      ↑
    详细设计 ←——————————→  集成测试
        ↓                  ↑
        编码 ←——————————→ 单元测试
              （开发）（测试）
```

### V模型各层对应关系

| 开发阶段 | 对应测试阶段 | 测试重点 | 负责人 |
|----------|-------------|----------|--------|
| 用户需求 | 验收测试（UAT） | 业务场景是否满足 | 客户/产品 |
| 系统需求 | 系统测试 | 端到端功能完整性 | 测试团队 |
| 概要设计 | 集成测试 | 模块间接口交互 | 测试/开发 |
| 详细设计 | 单元测试 | 函数/方法逻辑 | 开发 |
| 编码实现 | 代码审查 | 代码规范与逻辑 | 开发 |

**V模型优点**：
- 结构清晰，每个阶段有明确的验证对象
- 测试活动在开发早期就启动（测试计划）
- 便于追踪需求到测试的覆盖度（RTM）

**V模型局限**：
- 是线性模型，不适合需求频繁变更的项目
- 缺陷在测试阶段才能发现，修复成本较高

---

## 4. W 模型（双V模型）

W模型在V模型基础上演进，测试与开发**完全并行**：

```
需求 → 设计 → 编码 → 集成 → 验收
  ↕      ↕      ↕      ↕      ↕
测试需求→测试设计→单元测试→集成测试→系统测试
```

**核心思想**：测试人员在需求阶段就介入，同步参与每个开发阶段的评审与测试。

| 对比维度 | V模型 | W模型 |
|----------|-------|-------|
| 测试启动时间 | 编码后 | 需求阶段 |
| 缺陷发现时机 | 较晚 | 更早 |
| 成本 | 缺陷修复成本较高 | 早发现早修复，成本低 |
| 适用场景 | 需求稳定项目 | 需求变化、敏捷项目 |

---

## 5. 敏捷测试模型

在敏捷开发中，测试不再是独立阶段，而是融入每个 Sprint：

```
Sprint 计划 → 开发 + 测试并行 → Sprint 评审 → 回顾
     ↑                                              |
     ←——————————— 下一个 Sprint ←——————————————————
```

**敏捷测试四象限**（Brian Marick）：

|  | 面向业务 | 面向技术 |
|--|----------|----------|
| **支持团队** | Q1: 功能测试、故事测试 | Q2: 单元测试、组件测试 |
| **评价产品** | Q4: UAT、探索性测试 | Q3: 性能测试、安全测试 |

## 6. 黑盒测试 vs 白盒测试 vs 灰盒测试

### 6.1 三种测试方式对比

| 维度 | 黑盒测试 | 白盒测试 | 灰盒测试 |
|------|----------|----------|----------|
| **别名** | 功能测试、行为测试 | 结构测试、透明盒测试 | 半透明测试 |
| **测试视角** | 用户视角（外部行为） | 开发者视角（内部结构） | 兼顾内外 |
| **知识要求** | 不需了解代码 | 需要了解代码/架构 | 了解接口协议 |
| **测试依据** | 需求规格说明书 | 代码、设计文档 | 接口文档+部分代码 |
| **适用阶段** | 系统测试、UAT | 单元测试 | 集成测试 |
| **发现缺陷类型** | 功能缺陷、业务逻辑错误 | 逻辑错误、死代码、异常路径 | 接口缺陷、通信错误 |

### 6.2 黑盒测试方法

| 方法 | 核心思想 | 适用场景 |
|------|----------|----------|
| **等价类划分** | 将输入域划分为若干等价区间，每类取1个代表值 | 输入范围明确的功能 |
| **边界值分析** | 重点测试边界附近的值（min-1, min, min+1, max-1, max, max+1） | 有数值范围限制的字段 |
| **因果图/决策表** | 分析输入条件组合与输出的对应关系 | 多条件组合逻辑 |
| **场景法** | 模拟用户真实使用流程，构建正常/异常场景 | 业务流程类功能 |
| **错误推测法** | 基于经验猜测可能出错的地方 | 补充测试、探索性测试 |
| **正交实验法** | 用正交表减少测试用例数量，覆盖主要组合 | 多参数组合场景 |

### 6.3 白盒测试覆盖标准

从弱到强的覆盖层级：

```
语句覆盖 < 判定覆盖 < 条件覆盖 < 判定/条件覆盖 < 路径覆盖
```

| 覆盖类型 | 说明 | 覆盖率要求（参考） |
|----------|------|-------------------|
| **语句覆盖** | 每条可执行语句至少执行一次 | ≥80% |
| **判定覆盖** | 每个判断的真/假分支各执行一次 | ≥70% |
| **条件覆盖** | 每个判断中每个条件的真/假值各取一次 | ≥70% |
| **路径覆盖** | 覆盖程序中所有可能的执行路径 | 关键模块 100% |

## 7. 测试类型速查

| 测试类型 | 说明 | 典型工具 |
|----------|------|----------|
| **功能测试** | 验证功能是否符合需求 | 手工、Selenium |
| **接口测试** | 验证API输入输出 | Postman、pytest+requests |
| **性能测试** | 压力、负载、稳定性 | JMeter、Locust |
| **安全测试** | SQL注入、XSS等漏洞 | OWASP ZAP、Burp Suite |
| **兼容性测试** | 不同浏览器/OS/设备 | BrowserStack |
| **回归测试** | 修改后验证已有功能不受影响 | 自动化测试套件 |
| **探索性测试** | 无脚本、基于经验的自由测试 | 手工 |
| **冒烟测试** | 快速验证核心功能是否可用 | 核心用例子集 |

## 8. 缺陷管理

### 缺陷生命周期

```
New（新建）→ Assigned（分配）→ Open（处理中）→ Fixed（已修复）
                                                      ↓
Closed（关闭）← Verified（已验证）← Retest（待回归）
                   ↓
             Rejected（拒绝）/ Deferred（延期）/ Duplicate（重复）
```

### 缺陷严重等级

| 级别 | 名称 | 说明 | 示例 |
|------|------|------|------|
| P0 | 致命（Critical） | 系统崩溃、数据丢失 | 登录后系统宕机 |
| P1 | 严重（Major） | 主要功能不可用 | 支付失败 |
| P2 | 一般（Minor） | 次要功能异常 | 列表排序错误 |
| P3 | 轻微（Trivial） | UI问题、文字错误 | 按钮颜色偏差 |

## 9. 网络业务测试特点 🌐

> **Q**: 为什么网络业务测试特别重视接口测试？  
> **A**: 因为网络业务（如API网关、微服务）的核心交互就是HTTP接口，前后端分离架构下，接口是质量的第一道防线。

**网络业务测试重点**：
- **协议层**：HTTP/HTTPS、TCP/IP、WebSocket
- **接口层**：RESTful API、gRPC、消息队列
- **性能层**：高并发、低延迟、稳定性
- **安全层**：认证鉴权、数据加密、防注入

## 10. 今日思考 💡

| 问题 | 思考 |
|------|------|
| 什么时候用黑盒，什么时候用白盒？ | 功能验收用黑盒；代码质量/单元测试用白盒；日常接口测试是灰盒 |
| V模型在敏捷中还适用吗？ | V模型适合需求稳定项目；敏捷用迭代+W模型思想更合适 |
| 怎么证明测试充分了？ | 通过覆盖率指标（代码覆盖率、需求覆盖率）+ 缺陷趋势收敛 |
