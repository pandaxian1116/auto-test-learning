## 1. 为什么需要测试用例设计方法？

**核心矛盾**：输入空间无限大，测试资源有限。

- 穷举测试不可能（一个8位密码就有数亿种组合）
- 需要用**系统化方法**筛选出**最有价值**的测试点
- 好的测试用例设计 = 以最少用例覆盖最多缺陷

**四大黑盒测试设计方法**：

```
等价类划分  →  缩小输入范围
边界值分析  →  聚焦高风险点
场景法      →  覆盖业务流程
决策表      →  覆盖条件组合
```

## 2. 等价类划分（Equivalence Class Partitioning）

### 2.1 核心思想

将输入域划分为若干"等价区间"，**同一区间内任意一个值的测试效果相同**，只需取一个代表值。

### 2.2 有效等价类 vs 无效等价类

| 类型 | 说明 | 目的 |
|------|------|------|
| **有效等价类** | 符合需求规格的合法输入 | 验证功能正常工作 |
| **无效等价类** | 不符合需求规格的非法输入 | 验证系统容错能力 |

### 2.3 划分规则

| 输入条件类型 | 等价类划分方式 |
|-------------|---------------|
| 范围条件（a ≤ x ≤ b） | 1个有效类（a到b）+ 2个无效类（<a 和 >b） |
| 特定值（x = 5） | 1个有效类（=5）+ 1个无效类（≠5） |
| 集合条件（x ∈ {A,B,C}） | 1个有效类 + 1个无效类 |
| 布尔条件（true/false） | 1个有效类 + 1个无效类 |
| 格式条件（如手机号11位数字） | 合法格式为有效类；位数错误/含字母等为无效类 |

### 2.4 示例：用户年龄输入（18-60岁）

| 等价类 | 类型 | 代表值 |
|--------|------|--------|
| 18 ≤ age ≤ 60 | ✅ 有效 | 30 |
| age < 18 | ❌ 无效 | 10 |
| age > 60 | ❌ 无效 | 70 |
| 非数字（字母、符号） | ❌ 无效 | "abc" |
| 空值 | ❌ 无效 | "" |

## 3. 边界值分析（Boundary Value Analysis, BVA）

### 3.1 核心思想

**错误最容易出现在边界处**（差一错误 Off-by-one Error）。在等价类的**边界点附近**重点测试。

### 3.2 标准边界点公式

对于范围 [min, max]，测试以下7个点：

```
min-1  │  min  │  min+1  │  中间值  │  max-1  │  max  │  max+1
  ❌         ✅       ✅           ✅          ✅        ✅        ❌
```

| 测试点 | 说明 |
|--------|------|
| min - 1 | 刚好低于最小值（无效） |
| min | 最小有效值 |
| min + 1 | 略高于最小值 |
| 中间某值 | 典型有效值 |
| max - 1 | 略低于最大值 |
| max | 最大有效值 |
| max + 1 | 刚好超过最大值（无效） |

### 3.3 示例：密码长度（6-16位）

| 输入长度 | 测试点类型 | 预期结果 |
|----------|-----------|----------|
| 5位 | min-1（无效） | 提示"密码太短" |
| 6位 | min（有效边界） | 注册成功 |
| 7位 | min+1（有效） | 注册成功 |
| 11位 | 中间值（有效） | 注册成功 |
| 15位 | max-1（有效） | 注册成功 |
| 16位 | max（有效边界） | 注册成功 |
| 17位 | max+1（无效） | 提示"密码太长" |

### 3.4 等价类 + 边界值结合使用

**最佳实践**：先用等价类划分确定范围，再用边界值分析精确取点。

```
等价类划分 → 确定"哪些区间需要测试"
边界值分析 → 确定"每个区间取哪些值"
```

## 4. 场景法（Scenario-Based Testing）

### 4.1 核心思想

用户使用软件时按**流程行动**，测试应模拟真实业务场景，覆盖**基本流**和各种**备选流/异常流**。

### 4.2 基本概念

| 概念 | 说明 |
|------|------|
| **基本流（Happy Path）** | 功能正常执行的主线路径，无任何异常 |
| **备选流** | 功能正常但走了不同分支（如选择不同付款方式） |
| **异常流** | 出现错误时的处理路径（如余额不足、超时等） |

### 4.3 场景构建步骤

```
1. 识别基本流（正常成功路径）
2. 识别所有可能的分叉点（条件判断）
3. 为每个分叉点设计备选流和异常流
4. 组合流形成完整测试场景
```

### 4.4 示例：ATM取款场景

**基本流**：
```
插卡 → 输入密码 → 选择取款 → 输入金额 → 确认 → 取钱 → 取卡
```

**所有场景**：

| 场景编号 | 场景描述 | 流程组合 |
|----------|----------|----------|
| SC-01 | 正常取款 | 基本流 |
| SC-02 | 密码错误一次后重试成功 | 基本流 + 密码错误备选流 |
| SC-03 | 密码连续错误3次，卡被锁定 | 基本流 + 密码错误异常流 |
| SC-04 | 余额不足 | 基本流 + 余额不足异常流 |
| SC-05 | 取款金额超过单次限额 | 基本流 + 超限异常流 |
| SC-06 | ATM机现金不足 | 基本流 + 现金不足异常流 |

## 5. 决策表（Decision Table）

### 5.1 核心思想

当功能涉及**多个条件的组合**时，用决策表系统列出所有条件组合及对应的动作，防止遗漏。

### 5.2 决策表结构

```
          │ 规则1 │ 规则2 │ 规则3 │ 规则4
──────────┼───────┼───────┼───────┼───────
条件：    │       │       │       │
  条件1   │   Y   │   Y   │   N   │   N
  条件2   │   Y   │   N   │   Y   │   N
──────────┼───────┼───────┼───────┼───────
动作：    │       │       │       │
  动作1   │   ✓   │       │       │
  动作2   │       │   ✓   │   ✓   │
  动作3   │       │       │       │   ✓
```

规则数 = 2^n（n为条件数量）

### 5.3 示例：电商优惠券使用规则

**条件**：
- C1: 是否为会员？（Y/N）
- C2: 订单金额是否 ≥ 100元？（Y/N）
- C3: 是否持有有效优惠券？（Y/N）

**动作**：
- A1: 享受9折会员折扣
- A2: 满100减10
- A3: 使用优惠券抵扣
- A4: 无优惠，原价支付

| 条件/动作 | 规则1 | 规则2 | 规则3 | 规则4 | 规则5 | 规则6 | 规则7 | 规则8 |
|-----------|-------|-------|-------|-------|-------|-------|-------|-------|
| C1: 会员 | Y | Y | Y | Y | N | N | N | N |
| C2: ≥100 | Y | Y | N | N | Y | Y | N | N |
| C3: 有券 | Y | N | Y | N | Y | N | Y | N |
| A1: 9折 | ✓ | ✓ | ✓ | ✓ | | | | |
| A2: 满减 | ✓ | ✓ | | | ✓ | ✓ | | |
| A3: 用券 | ✓ | | ✓ | | ✓ | | ✓ | |
| A4: 原价 | | | | | | | | ✓ |

### 5.4 决策表适用场景

- 业务规则复杂，有多个条件组合
- 条件之间相互独立
- 需要保证不遗漏任何条件组合

## 6. 四种方法对比与选择

| 方法 | 适用场景 | 用例数量 | 主要缺陷类型 |
|------|----------|----------|-------------|
| 等价类划分 | 输入有明确范围/类别 | 较少 | 非法输入处理 |
| 边界值分析 | 数值型边界条件 | 适中 | 差一错误 |
| 场景法 | 业务流程、用户操作路径 | 较多 | 流程缺陷 |
| 决策表 | 多条件组合规则 | 视条件数量 | 逻辑遗漏 |

**实战建议**：
1. 先用**等价类**划分输入域
2. 用**边界值**精化取值点
3. 用**场景法**覆盖业务流程
4. 用**决策表**覆盖复杂业务规则
5. 用**错误推测法**补充经验性测试点

## 7. 测试用例模板

一个标准测试用例包含以下字段：

| 字段 | 说明 |
|------|------|
| 用例编号 | TC-模块-序号，如 TC-LOGIN-001 |
| 用例标题 | 简洁描述测试目的 |
| 所属模块 | 功能模块名 |
| 优先级 | P0/P1/P2/P3 |
| 前置条件 | 执行用例前必须满足的条件 |
| 测试步骤 | 具体操作步骤（有序） |
| 测试数据 | 具体的输入数据 |
| 预期结果 | 系统应该返回的正确结果 |
| 实际结果 | 执行后填写 |
| 测试状态 | Pass / Fail / Blocked |
| 设计方法 | 等价类/边界值/场景法/决策表 |

## 10. 今日思考 💡

| 问题 | 思考 |
|------|------|
| 等价类划分和边界值有什么区别？ | 等价类决定"测哪些区间"，边界值决定"每个区间取哪个值"，两者配合使用 |
| 什么时候用场景法而不是等价类？ | 当测试的是"用户行为流程"而非"单一输入框"时，用场景法 |
| 决策表适合什么规模的条件？ | 3-4个条件最合适；条件太多（>5个）用例数爆炸，需要精简 |
| 这10个用例够用吗？ | 基础覆盖足够，实际项目还需加：并发注册、SQL注入、XSS等安全用例 |
